# 爆款机制提炼与原创护栏指南

## 目的

本指南用于帮助 `webnovel-init` 和 `webnovel-plan` 在参考热门网文时，只提炼“机制层”的共性方法，而不滑向具体内容复用。

目标是：
- 学习爆款作品的结构机制
- 输出原创项目的初始化与规划输入
- 明确原创护栏，避免仿写

这不是仿写工具说明，不允许把参考作品拆成可直接照搬的人物、设定、桥段或表述。

## 核心原则

1. 只提炼跨作品共性机制，不提炼单书专属内容。
2. 只提炼结构、节奏、兑现、张力和升级模式，不复用故事材料。
3. 参考作品越多，越应抽象成“模式”，而不是“像哪一本”。
4. 若分析结果出现具体人物、具体反转、具体名场面，应立即回退并重写为机制层表达。
5. 创意约束优先于爆款参考。爆款机制只能增强原创方案，不能覆盖原创护栏。

## 允许提炼的维度

以下维度允许作为“机制层”输入：

- `market_mechanisms`
  - 读者为什么会持续追更
  - 题材组合如何形成稳定预期
  - 爽点、悬念、情绪兑现如何交替出现

- `opening_hooks`
  - 开篇如何快速立冲突
  - 开篇如何制造身份差、信息差、危机差
  - 前 1-3 章如何给出继续阅读理由

- `power_growth_patterns`
  - 主角成长节奏
  - 资源获取与代价机制
  - 阶段性升级、突破、反压制的设计规律

- `coolpoint_patterns`
  - 常见爽点类型组合
  - 小爽点、中爽点、大兑现的分布
  - 读者期待的反击、揭牌、碾压、逆转模式

- `chapter_end_hook_patterns`
  - 章末悬念、危机、承诺、选择、情绪钩的使用规律
  - 钩子与“章末未闭合问题”的配合方式

- `relationship_tension_patterns`
  - 角色关系张力的制造方式
  - 欲望错位、立场冲突、情感推拉、团队失衡等模式

## 禁止复用的维度

以下内容不得从参考作品中直接迁移：

- 角色姓名、身份组合、角色关系骨架
- 世界观核心装置、独特规则系统、核心外挂设定
- 关键反转、高潮桥段、结局兑现逻辑
- 标志性台词、专属表达、名场面包装
- 某本作品一眼可识别的叙事 gimmick

如果用户要求“做得像某一本”，必须改写为：
- 它吸引人的机制是什么
- 这些机制如何转译到一个全新原创项目中

## 建议输入格式

当用户提供参考作品时，优先收集：

- 参考作品名，建议 `3-5` 本
- 用户喜欢它们的原因
- 用户想借鉴的是哪一层：开篇、升级、爽点、悬念、感情线、群像、节奏

若参考作品少于 `3` 本，可以继续，但必须明确标记：

`样本不足，稳定性较低`

## 建议输出结构

统一使用以下结构块，并写入 `.webnovel/idea_bank.json`：

```json
{
  "bestseller_mechanism_profile": {
    "reference_titles": [],
    "market_mechanisms": [],
    "opening_hooks": [],
    "power_growth_patterns": [],
    "coolpoint_patterns": [],
    "chapter_end_hook_patterns": [],
    "relationship_tension_patterns": [],
    "originality_guardrails": []
  }
}
```

### 字段说明

- `reference_titles`
  - 仅记录书名，不记录“照搬建议”

- `market_mechanisms`
  - 提炼跨作品稳定有效的追读机制

- `opening_hooks`
  - 提炼适用于原创项目的开篇机制

- `power_growth_patterns`
  - 提炼成长和兑现的节奏规律

- `coolpoint_patterns`
  - 提炼爽点的结构分布与触发方式

- `chapter_end_hook_patterns`
  - 提炼章末钩子的常用模式

- `relationship_tension_patterns`
  - 提炼人物关系的张力组织方式

- `originality_guardrails`
  - 明确禁止直接复用的内容
  - 明确必须主动做差异化处理的部位

## 在 `webnovel-init` 中的用法

`webnovel-init` 应该：

1. 收集 `3-5` 本参考作品
2. 只提炼跨作品共性机制
3. 若分析出现具体情节复用，立即中断并重写
4. 生成 `bestseller_mechanism_profile`
5. 将其写入 `.webnovel/idea_bank.json`
6. 再进入创意约束包与卖点筛选

若用户不想提供参考作品：
- 允许跳过
- 必须显式标记“未启用爆款机制提炼”

## 在 `webnovel-plan` 中的用法

`webnovel-plan` 应该：

1. 检查 `.webnovel/idea_bank.json` 是否存在 `bestseller_mechanism_profile`
2. 若存在，则在卷纲和章纲中继承这些“机制模式”
3. 输出 `爆款机制继承` 说明本卷借用了哪些机制
4. 输出 `原创差异化校验` 说明本卷如何避免相似化

规划时可以继承：
- 节拍组织方式
- 爽点密度规律
- 章末钩子模式
- 关系张力组织方式

规划时不得继承：
- 某本作品的具体冲突链
- 某本作品的关键反转
- 某本作品的专属角色结构

## 原创护栏最小集合

建议至少包含以下规则：

1. 不直接复用参考作品的角色关系骨架。
2. 不直接复用参考作品的核心世界装置。
3. 不直接复用参考作品的关键反转、高潮桥段、结局兑现逻辑。
4. 不沿用参考作品的标志性台词、叙述包装或专属表达。
5. 若卷级核心冲突与某本参考作品高度相似，至少改写以下三项之一：
   - 冲突源
   - 代价机制
   - 胜负条件

## 失败信号

出现以下情况时，应视为失败并回退：

- 输出中大量出现某本书的具体剧情摘要
- 输出依赖某角色关系或某反转才能成立
- 输出方案去掉参考书名后就无法成立
- 输出文本像“换皮版本”而不是原创项目机制说明

## 一句话边界

允许学习爆款作品的机制，不允许复用爆款作品的内容。
