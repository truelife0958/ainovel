# Webnovel Writer 聚合 API 优先的远程化创作工作台设计

**Date:** 2026-04-10

**Supersedes:**
- `docs/superpowers/specs/2026-03-23-web-app-productization-design.md`
- `docs/superpowers/specs/2026-04-06-single-project-control-center-design.md`

## 背景

当前 Webnovel Writer 已经具备项目创建、立项、设定、大纲、写作、审查、模型设置等能力，但产品形态仍然被本地工作流强烈牵引：

- 一级导航仍然是 7 个并列页面，普通用户需要自己判断下一步去哪里
- 设置页默认暴露多 provider 卡片、多模型角色路由和大量技术字段，首次接入成本过高
- 设定、大纲、写作分散在不同页面，导致创作主链路频繁切页
- 当前系统大量使用目录、文件名、`state.json`、当前项目标记等本地存储心智
- 浏览器可用，但产品语义仍更像“本地项目管理壳”而不是“远程 AI 写作产品”

本次设计要解决的不是单个页面美化，而是重定义产品主语：

- 从目录、文件、路径、标记文件
- 切换为连接、项目、资料、章节、审查问题

## 已确认的产品决策

本设计基于以下已确认约束：

1. 同时支持聚合 API 与直连 provider
2. 默认体验优先引导聚合 API
3. 第一优先级是全链路体检与准产品级重构优化
4. 产品结构需要大幅收口
5. 需要尽量弱化一切本地依赖
6. 服务端托管是目标姿态
7. 允许破坏性调整
8. 后续必须用浏览器完成真实远程 API 的主链路验证

## 目标

本次设计的目标是把 Webnovel Writer 重构为一个聚合 API 优先、创作为核心、远程化优先的单用户写作产品。

具体目标：

- 将一级导航从 7 个并列页面收口为 5 个
- 将“连接 AI”从技术配置中心改造为面向普通用户的接入向导
- 将设定、大纲、写作合并为单一创作工作台
- 将审查页改造成问题中心和修复入口，而不是第二个编辑器
- 将前台主语从路径和文件切换为项目 ID、文档 ID、章节 ID
- 将本地文件存储降级为兼容层，而不是默认产品心智
- 保证真实远程 API 下的主链路可跑通、可测试、可验收

## 非目标

本次设计不包含以下内容：

- 多用户协作
- 登录、权限和团队空间
- 支付、订阅和计费体系
- 模板市场、插件市场
- 复杂工作流编排器
- 为普通用户暴露细粒度模型路由控制台
- 在第一版彻底删除所有本地存储实现

## 产品定位

重构后的 Webnovel Writer 不是“本地项目目录的网页壳”，而是一个以远程 AI 为默认能力、围绕作品推进的创作产品。

产品主链路应压缩为：

**建项目 → 连 AI → 定作品核心 → 在创作工作台完成资料、结构、章节 → 用审查闭环修复问题**

## 信息架构

一级导航收口为 5 个页面：

1. `项目`
2. `连接`
3. `立项`
4. `创作`
5. `审查`

### 1. 项目页

职责：

- 创建项目
- 切换项目
- 展示当前进度和最近工作
- 提供“继续创作”入口

页面原则：

- 前台不展示目录名、本地路径、项目根目录
- 普通用户不需要输入目录名，系统自动生成内部 slug 或 project id
- 项目卡只展示作品名、题材、阶段、当前卷章、最近修改时间等业务信息

### 2. 连接页

职责：

- 完成 AI 接入
- 明确当前生效连接
- 测试连接可用性

页面原则：

- 默认首屏只展示聚合 API 接入路径
- 聚合 API 是推荐选项，也是默认主流程
- 直连 provider 保留，但放入“高级配置”而不是首屏九宫格
- 默认只让用户完成 5 件事：选择连接方式、填写 Base URL、填写 API Key、选择模型套餐、测试连接并启用

### 3. 立项页

职责：

- 定义作品核心
- 产出统一作品卡
- 给创作页提供初始方向

页面原则：

- 主字段只保留题材、主角核心、金手指或核心机制、核心卖点、目标读者、故事钩子
- 次级字段折叠到“展开更多”
- 主按钮是“进入创作”，不是单纯“保存”

### 4. 创作页

职责：

- 统一承载设定、大纲、章节写作
- 成为产品默认主工作台
- 容纳 AI 规划、生成、修补动作

页面原则：

- 不再要求用户在设定、大纲、写作之间来回跳转
- 创作页内部按当前选中对象展示对应编辑形态
- 章节模式优先级最高，因为最终价值落在可推进的章节产出上

### 5. 审查页

职责：

- 展示问题列表
- 解释问题原因
- 把用户送回创作页完成修补

页面原则：

- 审查页不是第二个编辑器
- 它只做问题中心、风险看板和修复入口
- 每条问题都要说明“问题是什么、为什么算问题、去哪里修”

## 功能取舍

### 保留

- 项目创建与切换
- 立项核心信息
- 远程 API 接入能力
- 聚合 API 与直连 provider 双支持
- 设定、大纲、章节三类内容
- AI 规划、AI 写作、AI 修补
- 审查与诊断能力

### 合并

- `设定页 + 大纲页 + 写作页` → `创作页`
- `总览页 + 项目切换区` → `项目页`
- `审查建议 + 修补动作` → `创作页助手区闭环`

### 降级到高级模式

- 多 provider 同屏配置
- 按角色逐项填模型
- Base URL 细节调优
- API Key 批量管理
- 兼容层字段与内部 provider 细节

### 前台隐藏

- 目录名优先输入
- 本地路径
- `state.json`、`.webnovel` 等存储术语
- “当前项目根目录”等技术文案

### 删除

- 单独的设定导航
- 单独的大纲导航
- 首屏 provider 九宫格
- 让普通用户理解本地文件结构的必要性

## 页面与交互设计

## 连接页：接入向导而不是配置中心

连接页默认展示以下信息和动作：

1. 连接方式：`聚合 API（推荐）`
2. Base URL
3. API Key
4. 模型套餐：`省钱 / 平衡 / 质量`
5. `测试连接并启用`

高级配置区可折叠展示：

- OpenAI
- Anthropic
- Gemini
- DeepSeek
- 自定义 OpenAI 兼容接口
- 按角色覆写模型
- 多 provider 备用策略

页面输出的核心状态只有一项：

- 当前生效连接

普通用户不需要理解 provider 配置对象，也不需要同时维护多张卡片。

## 创作页：唯一主工作台

创作页采用三段式布局：

### 左侧：资料树

统一展示三类内容：

- 设定
- 大纲
- 章节

支持：

- 新建
- 搜索
- 最近访问
- 当前对象高亮
- 章节按卷 / 章分组

### 中间：主编辑区

根据当前选中对象切换：

- 选中设定：展示设定编辑器
- 选中大纲：展示大纲编辑器
- 选中章节：展示任务书与正文编辑器

章节模式采用“任务书在上、正文在下”的结构。

### 右侧：AI 助手区

右侧助手区根据对象类型变化，但始终保持少量高价值动作：

- 设定：补全、提炼、重组
- 大纲：扩写、细化节拍、补下一层结构
- 章节：规划本章、生成正文、续写、修补问题

每次只突出一个主按钮，其余动作进入次级菜单，避免按钮堆砌。

## 项目页：去掉本地目录感

项目页只展示业务对象，不展示底层路径信息。

创建项目时：

- 用户输入作品名、题材、目标读者等核心信息
- 系统自动生成内部 project id
- 目录名不再作为主流程输入项

项目列表中只显示：

- 作品名
- 题材
- 当前阶段
- 当前卷章
- 最近修改
- 继续创作入口

## 立项页：作品卡而不是散表单

立项页的目标是产出统一作品卡，而不是保存一堆松散字段。

主要输入：

- 题材 / 赛道
- 主角核心
- 金手指或核心机制
- 核心卖点
- 目标读者
- 故事钩子

立项完成后的主动作是进入创作页，而不是停留在立项页继续打磨低频信息。

## 审查页：问题中心

每条审查问题统一结构：

- 问题是什么
- 为什么是问题
- 去哪里修
- 推荐修补动作

主要动作：

- 查看问题
- 跳转创作页
- 自动带入修补建议
- 修完后重新审查

## 首次使用流程

首次用户链路必须压缩为：

1. 新建项目
2. 完成连接
3. 填写作品核心信息
4. 进入创作

系统不应要求首次用户先理解 provider 列表、文件结构、目录名或本地项目根路径。

## 保存与写回策略

### 手写内容

- 使用防抖自动保存
- 页面显示保存中 / 已保存 / 保存失败三态
- 自动保存不要求用户频繁手动点击保存按钮

### AI 生成内容

AI 生成、修补、改写等动作不能直接无提示覆盖正文。

统一写回方式：

- 预览结果
- 追加
- 替换
- 取消

这样可以避免一键毁稿，也更符合产品级写作体验。

## AI 状态统一规则

所有页面统一使用 4 种状态：

### 未连接

- 文案：`AI 未就绪，请先完成连接`
- 主动作：`去连接`

### 可执行

- 文案：`AI 已就绪`
- 显示当前套餐：省钱 / 平衡 / 质量
- 高级模式下可附带 provider 标签，但不喧宾夺主

### 执行中

- 文案明确到动作级别，如“正在规划本章”“正在生成正文”
- 禁止重复提交
- 长动作支持取消

### 执行失败

- 错误信息必须翻译成用户可理解的产品语言
- 不能直接把底层 provider 异常抛给用户

## 数据模型

前端主语统一为业务对象，而不是文件系统。

## 1. ConnectionProfile

表示当前生效的 AI 连接：

- `id`
- `mode`：`aggregated | direct`
- `provider`
- `baseUrl`
- `keyStatus`：`configured | missing`
- `qualityPreset`：`budget | balanced | quality`
- `advancedOverrides`
- `lastTestStatus`
- `lastTestedAt`

## 2. Project

表示一本书：

- `id`
- `title`
- `genre`
- `audience`
- `hook`
- `currentVolume`
- `currentChapter`
- `activeDocumentId`
- `status`：`ideation | drafting | reviewing`
- `updatedAt`

## 3. AssetDocument

统一表示设定与大纲：

- `id`
- `projectId`
- `type`：`setting | outline`
- `title`
- `content`
- `parentId`
- `sortOrder`
- `updatedAt`

## 4. Chapter

章节单独建模：

- `id`
- `projectId`
- `chapterNo`
- `title`
- `brief`
- `draft`
- `status`：`pending_plan | pending_draft | pending_review | completed`
- `wordCount`
- `updatedAt`

## 5. ReviewIssue

表示可追踪的审查问题：

- `id`
- `projectId`
- `chapterId`
- `severity`
- `category`
- `message`
- `suggestion`
- `targetType`
- `targetId`
- `status`：`open | resolved | ignored`

## 状态暴露原则

前台只暴露业务状态，不暴露技术状态。

允许展示：

- 当前连接是否可用
- 当前项目是什么
- 当前正在编辑什么
- 审查发现了什么问题
- AI 动作是否成功

不允许直接展示：

- 保存到 `state.json`
- 当前项目根目录
- 新建了哪个本地文件
- 当前 marker 文件或路径切换细节

## API 边界

接口应围绕资源，而不是围绕“当前项目 + 文件动作”。

## 连接接口

- `GET /api/connection`
- `PUT /api/connection`
- `POST /api/connection/test`

职责：

- 读取连接配置
- 保存连接配置
- 测试聚合 API 或直连 provider
- 返回当前连接可用性

## 项目接口

- `GET /api/projects`
- `POST /api/projects`
- `GET /api/projects/:projectId`
- `PATCH /api/projects/:projectId`
- `POST /api/projects/:projectId/select`

职责：

- 创建项目
- 更新项目
- 切换当前项目
- 返回项目摘要和最近工作状态

## 创作接口

- `GET /api/projects/:projectId/workspace`
- `POST /api/projects/:projectId/assets`
- `PATCH /api/projects/:projectId/assets/:assetId`
- `POST /api/projects/:projectId/chapters`
- `PATCH /api/projects/:projectId/chapters/:chapterId`

职责：

- 返回创作页资料树与当前工作对象
- 创建与更新设定、大纲、章节
- 让创作页一次性拿到工作台所需数据

## AI 动作接口

- `POST /api/projects/:projectId/chapters/:chapterId/plan`
- `POST /api/projects/:projectId/chapters/:chapterId/write`
- `POST /api/projects/:projectId/assets/:assetId/expand`
- `POST /api/projects/:projectId/review/:issueId/fix`

职责：

- 只接收业务 ID 与用户意图
- 不向页面暴露 provider 请求细节和文件路径
- 统一返回可预览、可追加、可替换的结果结构

## 审查接口

- `POST /api/projects/:projectId/review`
- `GET /api/projects/:projectId/review/issues`
- `PATCH /api/projects/:projectId/review/issues/:issueId`

职责：

- 发起审查
- 拉取问题
- 更新问题状态

## AI 调度层

需要单独的 AI Orchestrator 作为统一调度边界。

职责：

- 读取当前生效连接
- 根据“省钱 / 平衡 / 质量”选择默认模型
- 按动作类型组装上下文
- 调用聚合 API 或直连 provider
- 把不同 provider 的返回值整理为统一结果

页面不应直接知道：

- 当前走的是哪家 provider
- 每家 provider 的字段差异
- 不同模型协议差异

页面只需要知道：

- 当前动作是否可执行
- 当前正在执行什么
- 返回结果如何写回

## 去本地化策略

本次设计不要求第一步完全删除本地存储，而是把本地降为适配层。

### 第一层：先改产品心智

立即完成：

- 去路径
- 去目录名优先输入
- 去 `state.json` 提示
- 去“当前项目根目录”等技术文案
- 页面只传业务 ID

### 第二层：引入仓储抽象

建立统一仓储接口：

- `ConnectionRepository`
- `ProjectRepository`
- `DocumentRepository`
- `ChapterRepository`
- `ReviewRepository`

上层页面和 route handler 只依赖仓储，不直接访问文件系统。

### 第三层：远程存储成为默认

默认部署姿态：

- 结构化数据进入远程数据库
- 正文和大文本进入数据库或对象存储
- 连接配置进入服务端安全存储

本地文件仅保留两类用途：

1. 导入旧项目
2. 开发环境 fallback

## 当前项目选择

“当前项目”必须保留，但不能继续以 marker 文件心智暴露给用户。

设计要求：

- 接口支持显式 `projectId`
- 前端缓存当前项目选择
- 服务端使用 session 或 cookie 记住最近项目
- 在无用户体系阶段按单用户会话处理
- 后续增加账号体系时不需要重写接口形状

## 异常处理

只处理高频、真实、对主链路有影响的异常。

### API Key 无效

- 提示：`连接失败，API Key 无效或已失效`
- 动作：跳转连接页修改

### Base URL 不可达

- 提示：`连接失败，接口地址不可访问`
- 动作：提示检查地址格式并重新测试

### 模型不存在或当前路由不可用

- 提示：`当前模型不可用，请切换套餐或高级配置`

### 余额不足或配额受限

- 提示：`当前连接额度不足，暂时无法继续生成`

### 超时

- 提示：`请求超时，请重试`
- 不能丢失当前编辑内容

### AI 返回格式异常

- 提示：`本次生成结果不可用，请重试`
- 不允许把坏结构直接写入正文

核心原则：

- AI 失败不能导致当前稿子丢失
- AI 失败不能导致当前项目状态混乱
- AI 失败不能让页面卡死
- AI 失败后用户必须知道下一步去哪

## 验收与测试标准

## 浏览器主链路 E2E

至少覆盖以下链路：

### 1. 首次用户链路

- 新建项目
- 完成聚合 API 连接
- 完成立项
- 进入创作

### 2. 创作链路

- 新建设定
- 新建大纲
- 新建章节
- AI 规划本章
- AI 生成正文
- 选择追加或替换写回

### 3. 审查闭环

- 发起审查
- 查看问题列表
- 从问题跳回创作
- 执行修补
- 重新审查

### 4. 项目切换链路

- 创建第二个项目
- 切换项目
- 保持各自内容隔离
- 返回上次工作位置

### 5. 连接链路

- 修改连接
- 测试连接
- 连接失效后 AI 按钮状态同步变化

## 真实远程 API 验证

必须有一组真实远程 API 测试，而不是只靠 mock 数据。

最低要求：

- 聚合 API 默认路径跑通
- 直连 provider 高级路径跑通
- AI 规划、AI 写作、AI 修补至少各成功一次
- 错误状态提示符合产品预期

## 响应式验收

至少覆盖：

- 桌面宽屏
- 常见笔记本宽度
- 手机宽度约 393px

验收标准：

- 无横向溢出
- 主按钮始终可见
- 左侧资料树在小屏可折叠
- 右侧助手区在小屏收为抽屉或标签页
- 不允许两个编辑区把小屏同时挤爆

## 布局与文案验收

### 导航标准

- 一级导航不超过 5 个
- 每页只有 1 个主目标
- 当前页标题必须一眼能懂

### 页面密度标准

- 首屏只有 1 个主 CTA
- 同层级不同时出现 5 个以上并列主按钮
- 高级配置默认折叠
- 技术字段默认隐藏

### 文案标准

- 不出现路径、文件名、内部 JSON 名称
- 不出现“当前项目根目录”“保存到 state.json”之类技术提示
- 所有文案面向创作，不面向开发调试

### 行为标准

- AI 不可用时，明确告诉用户去哪里解决
- 审查发现问题时，明确告诉用户去哪里修
- 不让用户自己判断该去设定页、大纲页还是写作页

## 迁移原则

允许破坏性调整，但迁移顺序必须保证主链路持续可验证。

建议顺序：

1. 先改导航与页面语义
2. 再把接口从路径 / 当前项目逻辑转到资源 ID
3. 再把存储访问统一下沉到仓储层
4. 最后切远程默认存储
5. 提供一次性导入旧项目入口，而不是继续把本地项目当主工作流

## 完成标准

本轮重构完成后，必须同时满足：

- 默认体验是聚合 API 引导
- 前台不再暴露本地存储心智
- 设定 / 大纲 / 写作合并为单一创作工作台
- 审查与创作形成问题修补闭环
- 真实远程 API 能跑通主要按钮链路
- 桌面端与移动端都没有明显布局问题
- 核心文案统一、简洁、非技术化

以下任一情况出现，都视为未完成：

- 普通用户首次进入仍要理解 provider 九宫格
- 主链路仍需要在多个页面频繁来回跳转
- 保存反馈仍直接提到 `state.json`
- AI 报错后用户不知道下一步怎么做
- 小屏下仍出现明显拥挤、溢出或按钮难点

## 结论

本设计将 Webnovel Writer 的下一阶段产品形态明确为：

**一个聚合 API 优先、远程化优先、以创作工作台为核心、以审查闭环保证质量的单用户写作产品。**

本次重构的关键不是继续堆页面，而是统一产品主语、压缩主链路、隐藏技术细节，并让真实远程 AI 调用成为默认而自然的体验。
