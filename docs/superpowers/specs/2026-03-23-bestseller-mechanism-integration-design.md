# 爆款机制提炼与原创护栏集成设计

## 背景

`webnovel-writer` 目前已经具备创意约束、反套路规则、追读力约束和大纲继承能力，但缺少一个明确的流程来吸收热门网文的共性机制，同时防止用户把系统误用为“仿写工具”。

目标不是复刻某一本书，而是把爆款作品中的结构方法论转译成可复用、可继承、可约束的初始化与规划输入。

## 目标

- 在 `webnovel-init` 中增加“爆款机制提炼”步骤。
- 在 `webnovel-plan` 中继承这些机制并执行原创差异化校验。
- 新增统一参考文档，明确“可借鉴的机制”和“禁止复用的内容”。
- 复用现有 `.webnovel/idea_bank.json` 作为结构化承载，不引入新的 Python schema、数据库表或命令。

## 非目标

- 不新增独立命令，例如 `/webnovel-bestseller-map`。
- 不扩展 Python 数据模型、SQLite 表结构或 CLI 参数。
- 不接入联网榜单抓取、平台热点分析或自动外部检索。
- 不改写 `/webnovel-write` 的章节生成流程。

## 需求摘要

用户希望把“参考热门网文共性，创作类似爆款但保持原创”的思路融入现有工作流。

系统应支持：
- 用户提供 `3-5` 本参考作品。
- 系统只提炼“机制层”信息，而不是具体情节和设定。
- 提炼结果进入初始化产物，并被后续卷纲/章纲规划继承。
- 规划阶段必须输出“继承了什么机制”和“做了哪些差异化处理”。

系统必须避免：
- 鼓励直接仿写某本作品。
- 复用角色关系、关键反转、世界观装置、名场面或标志性表述。
- 让“爆款参考”覆盖现有创意约束系统和原创护栏。

## 核心设计

### 1. 新增统一参考文档

新增一份面向 `init` 与 `plan` 共用的参考文档，建议放在：

`/.claude/references/bestseller-mechanism-guide.md`

文档内容包含：
- 爆款机制提炼原则
- 允许提炼的维度
- 禁止复用的维度
- 结构化字段说明
- 在初始化和规划阶段的使用方法

该文档是统一事实源，避免在多个 skill 中复制规则。

### 2. `webnovel-init` 增加“爆款机制提炼”阶段

在现有 Deep 初始化流程中，新增一个阶段，位置放在“创意约束包”之前，作为创意约束的输入增强层。

新增阶段职责：
- 要求用户提供 `3-5` 本参考作品，可少于 `3` 本但需显式声明样本不足。
- 对每本作品只提炼机制，不记录具体桥段复用建议。
- 汇总出跨作品共性机制，而非逐本模仿策略。
- 基于共性机制生成“原创护栏”。
- 将结果写入 `.webnovel/idea_bank.json` 的新增结构块中。

新增阶段输出字段：
- `reference_titles`
- `market_mechanisms`
- `opening_hooks`
- `power_growth_patterns`
- `coolpoint_patterns`
- `chapter_end_hook_patterns`
- `relationship_tension_patterns`
- `originality_guardrails`

### 3. 复用 `.webnovel/idea_bank.json`

不新增独立存储文件，而是在现有创意产物中加入一个轻量结构块，例如：

```json
{
  "bestseller_mechanism_profile": {
    "reference_titles": [],
    "market_mechanisms": [],
    "opening_hooks": [],
    "power_growth_patterns": [],
    "coolpoint_patterns": [],
    "chapter_end_hook_patterns": [],
    "relationship_tension_patterns": [],
    "originality_guardrails": []
  }
}
```

这样可以保持与当前“创意约束包”一致的继承路径，避免改动脚本层。

### 4. `webnovel-plan` 继承机制画像

在卷纲与章纲规划阶段，如果 `.webnovel/idea_bank.json` 中存在 `bestseller_mechanism_profile`，则：

- 读取该结构块。
- 在卷级节拍、爽点密度、章末钩子、关系张力设计中优先参考“机制模式”。
- 明确区分“机制继承”和“内容原创”。

规划阶段新增两个强制输出块：

1. `爆款机制继承`
   - 本卷继承了哪些开篇钩子机制
   - 本卷继承了哪些升级/兑现模式
   - 本卷继承了哪些章末钩子与关系张力策略

2. `原创差异化校验`
   - 本卷与参考作品相比的关键差异点
   - 为避免相似化而主动替换的设定/冲突/角色关系
   - 哪些常见爆款套路被保留，哪些被反转或重构

### 5. 原创护栏规则

原创护栏是强约束，不是建议项。最小规则集：

- 不得直接复用参考作品的角色关系骨架。
- 不得直接复用参考作品的核心世界装置。
- 不得直接复用关键反转、高潮桥段、结局兑现逻辑。
- 不得沿用标志性台词、叙述包装或专属表达。
- 若某一卷的核心冲突与某本参考作品高度相似，必须改写冲突源、代价机制或胜负条件至少一项。

## 数据流

### 初始化阶段

用户输入参考作品列表
-> `webnovel-init` 提炼共性机制
-> 生成 `bestseller_mechanism_profile`
-> 写入 `.webnovel/idea_bank.json`

### 规划阶段

`webnovel-plan`
-> 读取 `.webnovel/idea_bank.json`
-> 继承机制画像
-> 生成卷纲/章纲
-> 输出“爆款机制继承”和“原创差异化校验”

## 方案对比

### 方案 A：仅增强提示词

优点：
- 改动最少
- 无需新增结构化产物

缺点：
- 后续无法稳定继承
- 容易退化成一次性对话提示

### 方案 B：结构化工作流集成

优点：
- 能进入现有 init -> plan 主链
- 有结构化承载，后续可继续扩展
- 改动范围可控

缺点：
- 需要同步修改两个 skill 和一份参考文档

### 方案 C：脚本层 schema 集成

优点：
- 最严格，最可校验

缺点：
- 当前阶段过重
- 与“先糅合进项目”的目标不匹配

推荐方案：`B`

## 组件边界

### `webnovel-init`

负责：
- 收集参考作品
- 提炼共性机制
- 生成原创护栏
- 写入结构化产物

不负责：
- 决定某一卷的具体继承策略

### `webnovel-plan`

负责：
- 继承机制画像
- 将机制落实到卷纲与章纲
- 输出原创差异化校验

不负责：
- 重做初始化阶段的参考作品分析

### `bestseller-mechanism-guide.md`

负责：
- 定义规则与字段
- 作为 init 与 plan 的统一规则来源

不负责：
- 存储项目数据

## 错误处理

- 若用户不给参考作品：允许跳过该阶段，但必须显式标注“未启用爆款机制提炼”。
- 若参考作品少于 `3` 本：允许继续，但输出中必须标记“样本不足，稳定性较低”。
- 若提炼结果落入具体情节复用：必须中断并回退到“机制层重写”。
- 若 `idea_bank.json` 缺少该结构块：`webnovel-plan` 按原流程继续，不报错。

## 验收标准

- `webnovel-init` 文档中存在“爆款机制提炼”流程说明。
- `webnovel-init` 明确限制只能提炼机制，不得复用具体内容。
- `webnovel-plan` 明确读取 `bestseller_mechanism_profile`。
- `webnovel-plan` 明确增加“爆款机制继承”和“原创差异化校验”输出要求。
- 新增一份统一参考文档，供 init 与 plan 共用。
- README 或项目指南中有一句明确说明：这是“机制借鉴 + 原创输出”，不是仿写工具。

## 测试与验证

### 文档级验证

- 检查新增参考文档路径是否可被两个 skill 正常引用。
- 检查两个 skill 的流程步骤是否引用同一规则源。
- 检查字段名在 `init` 与 `plan` 两端保持一致。

### 手工流程验证

1. 用 `3` 本参考作品执行 `/webnovel-init`
2. 确认输出中包含机制提炼和原创护栏
3. 确认 `.webnovel/idea_bank.json` 中出现 `bestseller_mechanism_profile`
4. 执行 `/webnovel-plan`
5. 确认卷纲或章纲中包含“爆款机制继承”和“原创差异化校验”

## 风险与缓解

- 风险：用户把“参考爆款”理解成“仿写模板”
  - 缓解：在参考文档、init、README 三处重复声明“仅提炼机制，不复用内容”

- 风险：字段过多导致初始化过重
  - 缓解：字段保持轻量，只保留对后续规划真正有用的模式字段

- 风险：后续 `/webnovel-write` 未直接读取该结构
  - 缓解：先通过 `plan` 阶段间接继承，不扩大本轮范围

## 实施步骤

1. 新增 `bestseller-mechanism-guide.md`
2. 修改 `webnovel-init` 的流程、引用和输出说明
3. 修改 `webnovel-plan` 的输入、流程和输出说明
4. 在 README 或 `CLAUDE.md` 增加简短说明
5. 做一次文档一致性检查
