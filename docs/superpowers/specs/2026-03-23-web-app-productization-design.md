# Webnovel Writer 可打包网页应用设计

## 背景

当前仓库本质上是一个基于 Claude Code 的长篇网文创作工作流包：

- 核心能力由 `.claude/skills/`、`.claude/agents/`、`.claude/scripts/` 提供
- 项目内容以文件系统和 `.webnovel/index.db` / `.webnovel/state.json` 为主
- 用户通过命令式工作流完成立项、规划、写作、审查与查询

现阶段的主要问题不是能力缺失，而是产品形态缺失：

- 普通作者无法直接以“应用”的方式使用这些能力
- 工作流分散在 skill、脚本和 Markdown 中，缺少统一可视入口
- 虽然底层已具备较强的长篇管理能力，但还不是可网页部署、可后续桌面打包的产品

## 目标

构建一个“个人作者优先”的一体化创作台，支持：

- 网页部署优先
- 后续桌面打包
- 从立项到大纲到章节写作到审查的一体化流程
- 用户自行填写 API Key，使用外部模型提供商
- 继续兼容现有项目目录、文件结构和 SQLite 存储

## 非目标

- 第一版不做团队协作优先设计
- 第一版不做模板市场
- 第一版不做发布平台一键发布
- 第一版不做自动识别全部伏笔/实体
- 第一版不把现有项目数据整体搬进新的中心数据库

## 目标用户

第一版服务对象：
- 个人作者

不把第一版做成：
- 编辑部协作平台
- SaaS 多租户产品
- 泛 AI 聊天应用

## 产品定位

这是一个“个人作者的一体化创作台”，而不是：

- 单纯的 AI 对话页面
- 单纯的 Markdown 编辑器
- 单纯的章节生成器

核心定位：
- 让作者围绕一个小说项目完成连续创作工作流
- 让 AI 成为工作流引擎，而不是唯一界面
- 把现有 skill/script 能力沉淀为可视化产品流程

## 核心模块

第一版产品模块划分为 7 个一级模块：

1. `项目首页`
2. `立项台`
3. `设定集`
4. `大纲台`
5. `章节写作台`
6. `审查与诊断中心`
7. `模型与项目设置`

### 1. 项目首页

职责：
- 显示当前项目概览
- 展示当前卷、当前章、总字数、最近活动
- 汇总待处理风险

主要内容：
- 当前项目状态
- 最近修改章节/设定
- 伏笔与审查提醒
- 待完成步骤入口

### 2. 立项台

职责：
- 把当前 `webnovel-init` 工作流可视化

主要内容：
- 基础信息
- 题材与目标读者
- 创意约束
- 爆款机制提炼
- 初始化草案生成

### 3. 设定集

职责：
- 管理设定与角色资料

主要内容：
- 世界观
- 力量体系
- 主角卡
- 主角组 / 女主卡
- 反派设计
- 实体关系资料

### 4. 大纲台

职责：
- 把 `webnovel-plan` 可视化

主要内容：
- 总纲
- 卷纲
- 节拍表
- 章纲
- 场景卡
- 伏笔卡入口

### 5. 章节写作台

职责：
- 提供当前章节的集中写作工作区

主要内容：
- 当前章任务书
- 写作编辑器
- 上章钩子 / 本章承接
- 目标 / 阻力 / 代价
- 本章相关实体、伏笔、节奏提醒
- 生成 / 改写 / 润色 / 续写操作

### 6. 审查与诊断中心

职责：
- 集中展示审查结果和长期健康度

主要内容：
- 一致性问题
- 节奏问题
- OOC 风险
- 追读力诊断
- 伏笔超期
- 质量趋势

### 7. 模型与项目设置

职责：
- 管理 API Key、模型路由和应用偏好

主要内容：
- 提供商设置
- 模型用途分配
- 成本模式
- 项目路径
- 导出与保存策略

## 信息架构

一级导航建议：

- 项目首页
- 立项
- 设定
- 大纲
- 写作
- 审查
- 设置

这套结构的原则：
- 单项目深度优先
- 让用户围绕一部小说连续工作
- 把现有工作流能力映射为固定页面，而不是临时命令

## 技术架构

推荐方案：`Web 主体 + Tauri 包装层`

### 推荐技术栈

- 前端：`Next.js`
- UI：React
- 服务层：`Next.js API Routes` / 同仓服务接口
- 数据层：
  - 项目数据继续使用文件系统 + SQLite
  - 应用配置单独存储
- 桌面包装：`Tauri`

### 为什么不选其他方案

不推荐 Electron 主导：
- 不符合“网页优先”
- 容易让 Web 版沦为次级形态
- 包体更重

不推荐纯 PWA：
- 不满足“可打包”的目标
- 后续桌面能力不足

### 架构原则

采用“应用壳 + 引擎层分离”：

- 应用壳：
  - 页面
  - 导航
  - 操作流程
  - 配置管理
  - 工作区体验

- 引擎层：
  - 状态逻辑
  - 文件读写
  - SQLite 逻辑
  - 写作/规划/审查服务

这样做的收益：
- 现有底层能力可以被逐步封装，而不是推倒重来
- Web 部署和桌面打包共享主体代码
- 后续若拆服务端，不会冲击前端结构

## 数据与运行方式

第一版采用“双数据面”。

### 1. 项目内容面

小说项目本身继续保留为目录结构：

- `设定集/*.md`
- `大纲/*.md`
- `正文/*.md`
- `.webnovel/state.json`
- `.webnovel/index.db`

应用不重新定义小说项目格式，而是直接管理现有项目目录。

### 2. 应用配置面

应用自身配置与缓存独立存储，例如：

- 最近打开项目
- 模型配置
- 提供商路由
- UI 偏好
- 最近调用日志

原则：
- 不污染小说项目内容
- 让同一个项目目录可在网页版和桌面版之间共享

## AI 接入方式

第一版采用“用户自填 API Key”的模式。

支持提供商：
- OpenAI
- Anthropic
- OpenRouter

建议每个工作流节点允许独立指定默认模型：
- 立项模型
- 大纲模型
- 写作模型
- 审查模型

## 成本控制

第一版建议内建基础成本控制机制：

- 每次调用前展示模型和用途
- 提供 `高质量 / 平衡 / 省钱` 三档
- 记录最近调用日志

成本控制不是可选附属，而是实际产品可用性的核心部分。

## 第一版交付边界

第一版定义为：

一个可部署、可演示、可实际开始写作的 Web 产品版本，并为桌面打包预留稳定通道。

### 第一版必须交付

- Web 应用可运行
- 可打开或创建小说项目目录
- 可完成以下 7 个页面的主流程：
  - 项目首页
  - 立项
  - 设定
  - 大纲
  - 写作
  - 审查
  - 设置
- 能读取和写回现有项目文件
- 能接入 OpenAI / Anthropic / OpenRouter 的 API Key
- 架构上可后续接 Tauri 打包

### 第一版允许简化

- 多项目管理只做最基本
- 模板市场不做
- 发布平台一键发布不做
- 自动识别全部伏笔/实体不做
- 高级协作不做

### 桌面打包策略

第一版建议：
- 架构上为 Tauri 预留
- 页面与接口按可打包要求设计
- 实际交付优先完成 Web 版
- 桌面壳验证放在 Web 版稳定之后

产品方向需博采众长，但不能机械拼贴。

### 借鉴点

  - 伏笔管理

  - 前置辅助工具整合

- `novelWriter` / `bibisco`

  - 长文项目结构化管理

### 转译原则

- 借鉴能力结构，不复制产品表面
- 优先增强“长篇创作工作流”而不是堆砌零散 AI 功能
- 以你现有项目底层能力为核心，而不是另起炉灶

## 风险与缓解

### 风险 1：产品范围过大

表现：
- 一次想做“写作器 + 编辑部系统 + 发布系统 + 协作平台”

缓解：
- 第一版只服务个人作者
- 严格限制到 7 个一级模块

### 风险 2：Web 版和桌面版相互牵制

表现：
- 为桌面能力牺牲 Web 部署稳定性

缓解：
- 明确 Web 优先
- 先做 Web 主产品，再补 Tauri 壳

### 风险 3：重做底层数据结构导致返工

表现：
- 一开始就改掉现有项目目录格式

缓解：
- 第一版继续兼容现有目录、文件和 SQLite
- 应用只是“管理项目”，而不是“吞掉项目”

### 风险 4：AI 能力强但产品不可用

表现：
- 有很多模型按钮，但没有稳定工作流

缓解：
- 页面设计围绕工作流，不围绕模型功能堆叠

## 验收标准

- 用户能在网页中完成项目创建/打开
- 用户能完成立项、设定、大纲、写作、审查的主流程
- 用户能配置 API Key 并完成至少一个工作流节点调用
- 应用能读写现有小说项目目录
- 页面结构与工作流一致，不需要依赖命令行才能正常使用
- 架构可后续接入 Tauri 打包
