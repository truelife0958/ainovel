# Webnovel Writer 全项目质量审计与修复设计

**Date:** 2026-04-14

## 背景

当前 Webnovel Writer 已经具备前端页面、API 路由、AI 动作、项目文档读写、单元测试和 Playwright E2E 基础能力，但用户明确反馈“有很多问题”，并要求从普通用户视角对前端、后端和整条链路做满强度测试、找出 BUG、清理无关文件、保持代码纯净，并同步梳理还能继续优化、完善和补强的地方。

现阶段最大风险不是某一个已知缺陷，而是缺少一轮基于真实产品使用路径的系统性体检。如果直接进入边查边修，很容易把表层问题修掉，却遗漏更深层的结构问题、文档失真、测试缺口和跨层断点。

因此，本次工作先进入“全面诊断优先”阶段：先建立质量基线、完成自动化验证与浏览器手测、按证据输出问题清单和优先级，再进入修复和重构。

## 已确认决策

本设计基于以下已确认选择：

1. 本轮第一阶段交付为全面诊断报告，不是边查边修
2. AI 功能验证采用“本地优先”模式
3. 不要求本阶段依赖真实 API Key 完成所有诊断
4. 需要覆盖前端、后端、工程质量、文档和用户体验
5. 测试视角以普通用户主链路为主，同时补充异常路径和工程链路
6. 问题必须先做根因分析，再决定是否修复

## 目标

本次设计的目标是为 Webnovel Writer 建立一套可执行的全项目质量审计方案，输出可直接指导后续修复与优化的诊断结果。

具体目标：

- 识别阻断普通用户使用的前端、后端和全链路 BUG
- 验证主要页面、路由和工作流是否可达、可用、可恢复
- 识别构建、测试、运行时和状态同步问题
- 找出 README 与 docs 和当前实现不一致的地方
- 标记无关文件、脏文件、可删资产和潜在死代码
- 给出修复优先级、清理建议、优化建议和功能补全建议

## 非目标

本次设计不包含以下内容：

- 在诊断阶段直接完成所有修复
- 在未定位根因前进行大规模重构
- 依赖真实远程 AI 提供商完成全部主链路验收
- 为未来多用户、计费、权限体系提前设计新架构
- 做与本轮质量审计无关的美化式改动

## 审计原则

### 1. 诊断先于修复

任何问题先记录复现路径、错误表现、影响范围和根因线索，再进入修复决策。禁止在没有证据的情况下靠猜测改代码。

### 2. 用户旅程优先

审计以“普通用户怎么走主流程”为主线，而不是只按技术层分模块阅读代码。先看用户能不能完成任务，再看底层为什么失败。

### 3. 证据驱动

构建报错、测试输出、浏览器操作结果、接口返回、日志和代码位置都要作为诊断依据，避免只写抽象判断。

### 4. 纯净优先

只保留当前产品交付真正需要的文件、文档和代码。把无关资产、脏文件、过时文档和冗余实现单独列出，避免继续污染仓库。

### 5. 分层归因

每个问题都尽量归到具体层级：页面层、组件层、状态层、API 层、存储层、测试层、文档层。跨层问题也要标明断点发生在哪里。

## 审计范围

本轮质量审计覆盖 5 个层面。

### A. 用户旅程审计

围绕普通用户实际使用路径检查：

- 进入首页
- 创建和切换项目
- 进入立项和资料编辑
- 进入创作工作台
- 新建设定、大纲、章节
- 保存内容和切换内容
- 调用 AI 动作入口
- 查看审查页面
- 进入设置与连接相关流程
- 遇到空状态、错误态、无配置态时能否继续

### B. 前端审计

检查内容包括：

- 页面可访问性与路由跳转
- 组件交互是否符合预期
- 加载、空状态、错误状态是否完整
- URL 参数与初始状态是否一致
- 保存反馈是否可信
- 页面布局在桌面与窄屏下是否稳定
- 是否存在明显误导文案、不可点击区域、状态错乱或交互死角

### C. 后端与 API 审计

检查内容包括：

- API 路由输入校验是否足够
- 错误处理和返回结构是否一致
- 前端请求参数与后端契约是否匹配
- 文件系统读写是否存在边界问题
- 限流、安全过滤、错误脱敏是否真实生效
- 发生失败时是否会破坏现有项目数据

### D. 工程与测试审计

检查内容包括：

- `npm run build` 是否通过
- 单元测试是否真实覆盖核心能力
- E2E 是否可运行、是否失真
- 测试脚本与仓库实际结构是否一致
- 是否存在测试缺口、脆弱测试、过期测试
- 是否存在无关文件、脏文件、临时文件、未纳入交付但污染仓库的内容

### E. 文档与交付一致性审计

检查内容包括：

- README 中的功能描述是否和当前产品一致
- docs 中的交付基线是否反映当前实际实现
- 启动、测试、页面、API 和工作流说明是否过时
- 是否需要补一份面向当前 Web App 的真实说明文档

## 测试与诊断方法

### 第一阶段：静态与结构检查

目标是快速建立项目基线，不执行修复。

检查内容：

- 目录结构、页面入口、API 路由、关键组件
- README、docs、测试脚本、运行脚本
- 当前仓库中明显异常的无关文件与脏文件
- 前端页面与后端路由的映射关系

产出：

- 审计对象清单
- 高风险区域初步标记
- 文档与实现偏差的第一批线索

### 第二阶段：自动化基线验证

目标是确认工程是否在当前状态下能构建、能跑测试、能支持后续手测。

执行内容：

- 运行生产构建
- 运行单元测试
- 运行 E2E / smoke 测试
- 记录所有失败项、报错信息和初步归因方向

产出：

- 构建基线结论
- 自动化测试失败清单
- 需要进入根因分析的问题集合

### 第三阶段：浏览器高强度手测

目标是从普通用户视角验证主流程和异常路径。

手测会覆盖：

- 首页与入口页是否清晰
- 项目创建 / 切换 / 继续操作是否顺畅
- 工作台文档选择、切换、保存、创建是否稳定
- AI 按钮在未连接和可连接状态下的反馈是否正确
- 审查与设置页面是否存在断链、空白、无反馈或错误文案
- 异常路径下页面是否可恢复，是否存在状态残留

产出：

- 用户旅程问题清单
- 复现步骤
- 截图或页面证据（如需要）
- 可见体验缺陷列表

### 第四阶段：根因分析

目标是对每个关键问题完成最少必要的证据收集。

方法：

- 对照构建输出、测试输出和浏览器复现结果定位断点
- 阅读相关页面、组件、API 和库代码
- 在需要时追踪状态流与请求流
- 区分表象问题与根因问题
- 将跨层问题拆解为起点、传播路径和最终表现

产出：

- 每个关键问题的根因判断
- 影响范围
- 修复难度和修复批次建议

### 第五阶段：诊断报告与后续建议

目标是把所有发现整理成可直接执行的结果，而不是散乱结论。

输出结构：

- 阻断级问题
- 高优先级问题
- 中优先级问题
- 低优先级问题
- 仓库清理建议
- 文档修正建议
- 结构优化建议
- 功能补全建议

## 测试矩阵

### 1. 页面与用户流程矩阵

至少覆盖以下页面或入口：

- `/`
- `/projects`
- `/ideation`
- `/review`
- `/connection`
- 其他从导航可达的重要入口

每个页面至少检查：

- 是否可打开
- 是否有明确主目标
- 是否有空状态 / 错误态
- 主操作是否可点击
- 状态切换后是否反馈正确

### 2. 创作工作台矩阵

重点验证：

- 三类文档切换：设定 / 大纲 / 章节
- 新建文档
- 编辑文档
- 保存文档
- 切换文档后的内容同步
- 任务书面板打开 / 保存 / 校验
- AI 动作入口在不同状态下的行为

### 3. API 与数据流矩阵

重点验证：

- 项目读取与当前项目切换
- 文档读取 / 创建 / 保存
- 任务书读取 / 保存
- 上下文读取
- AI 动作路由的校验、限流、失败返回
- 设置相关接口的读取与保存

### 4. 工程链路矩阵

重点验证：

- 开发态能否启动
- 生产构建是否通过
- 单测是否通过
- E2E 是否可运行
- 脚本与仓库当前布局是否一致

### 5. 文档与仓库卫生矩阵

重点验证：

- README 描述是否真实
- docs 导航是否指向当前有效文档
- 是否存在临时文件、残留文件、无关目录
- 是否存在应删除但仍在仓库中的历史资产

## 问题分级标准

### P0 / 阻断

普通用户无法完成核心操作，或系统存在明显数据风险。

例如：

- 关键页面打不开
- 核心保存失败
- 项目无法创建或切换
- 一进入主流程就报错
- 错误导致内容丢失或状态严重错乱

### P1 / 高优先级

主流程可以进入，但体验严重受损，或明显偏离产品预期。

例如：

- 关键按钮无效
- 页面状态错误
- API 契约不一致导致功能不稳定
- 文档严重误导启动或测试方式

### P2 / 中优先级

不阻断主流程，但持续影响效率、可维护性或可信度。

例如：

- 空状态和错误态不完整
- 局部布局问题
- 部分测试失真
- 存在较明显冗余代码或无关文件

### P3 / 低优先级

优化项、可用性打磨项、未来迭代项。

例如：

- 文案不统一
- 小范围交互粗糙
- 可补强但不影响当前交付的问题

## 诊断报告格式

最终诊断报告中的每个问题至少包含：

- 标题
- 严重级别
- 所属层级
- 复现步骤
- 实际表现
- 预期表现
- 根因判断
- 影响范围
- 建议修复方式
- 涉及文件或路由

## 后续修复批次建议

诊断完成后，后续修复建议按 4 批推进：

### 批次 1：阻断主流程

先解决 P0 和最关键的 P1，确保产品能被普通用户使用。

### 批次 2：稳定核心链路

解决页面状态、API 契约、保存反馈、主工作台交互等高频问题。

### 批次 3：清理与纯净化

删除无关文件、冗余资产、过期文档，收敛仓库噪音，保持代码和文档一致。

### 批次 4：优化与补完

处理中低优先级体验问题、测试缺口、功能补全和结构优化。

## 风险与约束

本设计明确以下约束：

- 诊断阶段不承诺直接修完所有问题
- AI 真实远程联调不作为本阶段完成标准
- 不能因为追求“快修”而跳过根因分析
- 不能把当前仓库中的异常文件直接删除了事，必须先确认其性质
- 文档更新必须以当前实际实现为准，不能延续失真的旧表述

## 完成标准

本轮“全面诊断优先”阶段完成后，必须同时满足：

- 已完成自动化基线验证
- 已完成主要用户旅程的浏览器手测
- 已输出按优先级排序的问题清单
- 已给出每个关键问题的根因判断或根因线索
- 已给出仓库清理建议
- 已给出 README / docs 偏差清单
- 已给出后续修复批次建议
- 结论足够支撑下一阶段进入系统修复

以下任一情况出现，都视为未完成：

- 只列出表象问题，没有复现与根因信息
- 没跑构建或测试就给结论
- 没做浏览器主流程验证就声称“整体可用”
- 没区分阻断级与一般问题
- 没检查 README / docs 就建议文档更新

## 结论

本设计将本轮工作定义为：

**一次以普通用户主流程为中心、以自动化验证和根因分析为支撑、覆盖前端、后端、工程质量、文档一致性和仓库纯净度的全项目质量审计。**

诊断结果将作为后续修复、重构、清理和文档更新的唯一入口，确保下一阶段改动有证据、有优先级、有边界。
